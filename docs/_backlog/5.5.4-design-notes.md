## 3. server action 설계

### 시그니처

```ts
'use server'
import { z } from 'zod'

const InputSchema = z.object({
  project_id: z.string().uuid(),
  file: z.instanceof(File),
})

export type UploadPhotoResult =
  | { ok: true; photoId: string; storagePath: string; signedUrl: string }
  | { ok: false; error: string }

export async function uploadPhotoAction(
  formData: FormData
): Promise<UploadPhotoResult>
```

### 처리 흐름

1. FormData parse + InputSchema 검증 → fail 시 `{ ok: false, error: 'validation' }`
2. `supabase = await createClient()` (server.ts)
3. `auth.getUser()` → `!user` 시 `{ ok: false, error: 'unauthorized' }`
4. `photoUuid = crypto.randomUUID()`
5. `storagePath = ${user.id}/${project_id}/${photoUuid}.jpg`
6. `storage.from('photos').upload(storagePath, file, { upsert: false })` → fail 시 `{ ok: false, error: 'storage' }`
7. photos INSERT (`id` = photoUuid, `project_id`, `uploader_id` = user.id, `storage_path`, …)
   - fail 시: `storage.from('photos').remove([storagePath])` rollback
   - rollback 결과 무관하게 `{ ok: false, error: 'db' }` (rollback 실패 시 `console.error` 로 orphan 로그)
8. `createSignedUrl(storagePath, 60)` → fail 시 `signedUrl: ''` 유지 (정합 위협 없음)
9. `{ ok: true, photoId: photoUuid, storagePath, signedUrl }`

### 정합 정책

**storage first + storage.remove rollback**.

- 근거: 현 코드 흐름과 동일, 파일 실 업로드 검증 후 DB 기록.
- orphan 가능성 (storage 파일 + DB row 부재): V2 cleanup job 백로그.
